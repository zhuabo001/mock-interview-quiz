⏺ 基于 P0 的交付范围（四张表的 CRUD + 筛选搜索 + 双向关联 + 2 个图表 + Vercel
  部署），给出我的推荐。

  1. Mock 层：MSW（不选 json-server）

  ┌──────────────┬───────────────────────────┬──────────────────────────────┐
  │     维度     │            MSW            │         json-server          │
  ├──────────────┼───────────────────────────┼──────────────────────────────┤
  │ 运行形态     │ 浏览器 Service Worker     │ 独立 Node 进程 + 端口        │
  │              │ 拦截                      │                              │
  ├──────────────┼───────────────────────────┼──────────────────────────────┤
  │ Vercel 部署  │ 纯静态，无额外配置        │ 需改成 Serverless            │
  │              │                           │ Functions，额外折腾          │
  ├──────────────┼───────────────────────────┼──────────────────────────────┤
  │ 复杂业务逻辑 │ 任意 JS，随便写聚合/连表  │ RESTful 规范，复杂查询要写   │
  │              │                           │ middleware                   │
  ├──────────────┼───────────────────────────┼──────────────────────────────┤
  │ 多模块复用   │ P1 的 Mock LLM            │ Agent 这种非 CRUD 场景不好塞 │
  │              │ 也能用同一套              │                              │
  ├──────────────┼───────────────────────────┼──────────────────────────────┤
  │ 团队体感     │ "真实网络请求"，DevTools  │ 也可见，但本地跨端口略麻烦   │
  │              │ 可见                      │                              │
  └──────────────┴───────────────────────────┴──────────────────────────────┘

  关键理由：文档明确要求 Day 4 必须在 Vercel 跑通 Demo，而 Vercel
  默认是静态托管。json-server 得额外搭 Serverless 或换平台，为了个 Mock
  得不偿失。MSW 打包后就是静态资源，pnpm build 完直接 vercel deploy
  结束战斗。另外 P1 的 AI Agent 方向（智能查询/辅助录入）也可以复用 MSW 做 LLM
  mock，一套基础设施打通两个需求。

  唯一要注意的：MSW 在生产环境启用 Service Worker 要主动声明（通常用
  VITE_USE_MOCK=true 开关），别让它在真实后端环境误伤。

  2. 状态管理：TanStack Query + Zustand

  这是目前最成熟的"服务端状态 / 客户端状态分离"组合：

  TanStack Query（服务端状态）
  - P0 里 90% 的状态都是远程数据：产品列表、客户列表、持仓、跟进记录
  - 自带缓存、去重、失效、乐观更新——「双向关联」场景下新增跟进记录后自动让客户详
  情和产品详情一起失效，一行 invalidateQueries 解决
  - Loading / Error / Empty 三态（P1 要求）是内置的 isLoading / isError / data
  字段，基本不用自己管
  - 和 MSW 配合天然顺滑，拦截的就是它发的 fetch

  Zustand（客户端状态）
  - 只管那一点点 UI 状态：全局筛选条件、侧栏开合、当前选中的客户 ID 等
  - API 比 Redux Toolkit 少一半，4 天项目不想写 slice + reducer + action
  三层仪式
  - 无需 Provider 包裹，Store 直接 import 就能用

  为什么不用 Redux Toolkit + RTK Query？
  RTK Query 功能上等价于 TanStack Query，但绑定了 Redux 的心智负担。题目是打磨
  MVP，不是演示企业级架构，选更轻的组合把时间省下来给业务逻辑和 DESIGN.md。

  为什么不用 Context + useReducer 手搓？
  持仓的双向查询、Dashboard
  的聚合数据会涉及跨页面缓存共享，手搓缓存失效策略是性价比最低的选择。

  3. UI 组件库：Ant Design + @ant-design/charts

  Ant Design
  - 基金销售系统本质是 B 端后台，Ant Design 是这个赛道的标准答案
  - Table 内置排序/筛选/分页/空态，产品货架和客户列表一把梭
  - Form + Modal/Drawer 把跟进记录新增表单直接配齐，省掉表单库
  - 中文语境原生，文案规范贴合题目场景（基金名、客户、金额）
  - 文档里点名的两个候选之一，稳

  为什么不用 shadcn/ui？
  美学更现代，但需要自己组装 Table 的分页/排序/筛选，Form 也要配 react-hook-form
   + zod。4 天项目里这部分组装时间是纯损耗，拿不到加分。如果题目是 C
  端产品展示，shadcn 会更合适。

  @ant-design/charts（Dashboard 两张图）
  - 和 Ant Design 同厂出品，主题、字体、间距天然一致
  - 饼图、折线图、柱状图 API 极简，写两张图不超过 100 行
  - 底层 G2Plot 成熟，应付题目要求的「规模占比 / 跟进趋势」绰绰有余

  备选：如果你对 ECharts 更熟，换成 echarts-for-react
  也完全可以，但要自己调一下色板匹配 AntD 主题。

⏺ 针对你这个 4 天 MVP 的场景，Vercel 部署的前置动作其实非常轻。

  一、前置动作清单（按顺序）

  1. 账号注册（2 个，都免费）

  - GitHub 账号（如果还没有）：代码托管用。Vercel 最顺滑的部署方式是绑定 Git
  仓库，push 即自动部署
  - Vercel 账号：直接用 GitHub OAuth 登录就行，不用单独注册。选 Hobby（个人版）
  计划

  2. 代码侧准备

  - 项目推到 GitHub 仓库（public 或 private 都可以，Hobby 计划都支持）
  - 确保 package.json 里有 build 脚本（Vite 默认就是 vite build）
  - 产物目录是 dist（Vite 默认），Vercel 会自动识别 Vite 项目
  - MSW 的 Service Worker 文件（public/mockServiceWorker.js）要提交到 Git，别被
  .gitignore 掉——这是最容易踩的坑
  - 根目录加个 vercel.json（可选）处理 SPA 路由回退：
  {
    "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
  }

  3. 部署动作

  - 在 Vercel 控制台点 "Import Project" → 选你的 GitHub 仓库 → 基本一路 Next
  - 首次部署完成后会得到一个形如 your-project-xxx.vercel.app 的地址，这就是交付 
  Demo 的 URL

  ---
  二、需要申请 API Key 或域名吗？

  都不需要。

  - API Key：Vercel 自身不需要。你的项目是纯前端 + MSW，没有外部服务依赖。只有
  P1 阶段如果接真实 LLM 才需要申请 OpenAI/Anthropic key
  - 域名：默认给 *.vercel.app
  二级域名，演示完全够用。自定义域名才需要花钱买域名（年费 ~$10-15），这次 MVP
  用不着

  ---
  三、收费情况

  你这个场景：0 元，且不会被扣费。

  Hobby 计划（免费版）限额：
  - 100 GB 带宽/月 — 一个演示级项目用不掉 1%
  - 6000 分钟构建时间/月 — Vite 构建一次几十秒，随便推
  - 商用限制 — Hobby 计划不允许商业用途，个人项目 / 作品集 / 面试 Demo 完全 OK

  超限会怎样？
  Hobby 计划不是「超了就扣钱」，而是超限后暂停服务，等 30 
  天周期重置。这意味着就算代码有 bug 被流量打爆，最坏结果也只是 Demo
  临时挂掉，不会产生任何费用——这对面试交付反而是好事，钱包绝对安全。

  什么时候才需要付费？
  - 要商业化 → Pro 计划 $20/月/人
  - 团队协作需要 seat → Pro
  - 需要密码保护预览、自定义分析 → Pro

  都与你当前需求无关。

  ---
  四、交付前的最后一公里建议

  题目文档里专门警告「Day 4 部署挂了基本等于没交」，所以：

  1. 别等到 Day 4 才第一次部署。Day 1 骨架搭完就先推一次，确认 CI 通路畅通
  2. MSW 在生产环境的启用开关要测一次。本地 dev 用 Vite proxy 没事，部署上去如果
   Service Worker 没注册，所有接口会 404
  3. 录屏兜底。即便 Vercel 正常，网络抽风也可能影响面试官访问，本地录个 2-3
  分钟的演示视频塞进交付物里

  ---
  各自优劣

  方案 A（docs/data-structure/design.md）

  优势：
  - SalesPerson 实体更贴近真实业务：CRM
  系统里业务员是核心角色，跟进记录归属到人、客户分配到人，这在真实场景中是必须的
  。如果面试官追问「多人协作怎么办」，有这张表就能自圆其说
  - 数据样例直观：直接给了 JSON 示例，阅读门槛低，适合和非技术人员对齐
  - establishNav 字段有业务价值：成立净值 = 1.0000
  是行业惯例，配合当前净值可以直接算产品成立以来的总回报

  劣势：
  - 只有模型没有接口：前端开发时还需要再设计一遍「怎么查、怎么聚合、怎么分页」，
  等于只做了一半
  - 没有考虑聚合场景：双向关联在数据层定义了，但没有说明前端怎么拿到这个关联数据
  （是一次请求还是多次？返回什么结构？）
  - SalesPerson 在 P0 阶段是负担：题目没有多用户/权限需求，引入业务员实体意味着
  Mock 数据要多维护一张表、前端要多处理一个关联，但 UI 上可能根本不展示

  方案 B（我的设计）

  优势：
  - 从接口消费者视角设计：直接回答了「前端调什么接口、传什么参数、拿到什么结构」
  ，MSW handler 可以照着写
  - 聚合接口减少前端复杂度：客户详情一个请求拿到 holdings +
  followups，不用管理多个 loading 状态
  - TypeScript 类型可直接落地：类型检查已通过，开发时有完整的类型提示
  - Dashboard 数据源明确：不用前端拿全量数据自己算，模拟了真实后端聚合行为

  劣势：
  - 砍掉 SalesPerson 
  是一个取舍：如果面试官问「这个系统多人用怎么办」，需要额外解释为什么 P0 不做
  - 缺少 establishNav：无法直接展示「产品成立以来总回报」，需要额外计算或补字段
  - 聚合接口耦合度高：客户详情返回 holdings +
  followups，如果后续只想单独刷新跟进列表（比如新增后），要么 invalidate
  整个详情查询，要么再加一个独立的 followups 列表接口（实际上已经加了 GET 
  /api/followups，所以这个问题不大）

  ---
  一句话总结

  方案 A 是数据建模视角，回答了「业务里有哪些东西、它们之间什么关系」；方案 B
  是接口消费视角，回答了「前端怎么拿数据、拿到什么形状的数据」。两者互补而非互斥
  ——最终落地时需要两者兼备。

     ---
     设计决策说明

     为什么 SalesPerson 不做独立的 CRUD 接口？

     P0 阶段业务员数据是只读的种子数据（Mock 里写死 2-3
     个人），不需要增删改。它的作用是：
     - 客户详情页展示「谁在负责这个客户」
     - 跟进记录标注「谁做的跟进」
     - 面试时能回答「多人协作场景下数据归属」的问题

     如果 P2 要做权限/多角色，再加 GET /api/salespersons 列表接口即可。

     为什么把 recentFollowUps 从客户详情中拆出去？

     原设计的初衷是「一次请求拿全」减少 loading 状态，但实际开发中：
     - 新增跟进是高频操作，每次新增后 invalidate 整个客户详情（含 holdings
     查询）代价过大
     - TanStack Query 的 useQuery 天然支持并行请求，客户详情页同时发
     clientDetail + followups 两个请求，用户感知不到差异
     - 拆开后两个缓存独立失效，粒度更细，体验更好

     为什么加 totalReturn 而不是让前端自己算？

     (nav - establishNav) / establishNav 虽然简单，但：
     - 如果多处展示（列表、详情、Dashboard），每处都算一遍是重复逻辑
     - Mock 数据里直接给好，前端零计算，也模拟了真实后端会做的事
