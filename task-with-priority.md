# 笔试题 MVP 优先级规划

> 核心判断依据：评分权重中「业务契合度 ★★★★★」最高，且题目明确强调"一个打磨到位的核心功能 > 五个粗糙的半成品"。因此 MVP 策略是：**三个模块都覆盖，但每个模块只做最能体现业务理解的那一刀**。

---

## P0：MVP 必须完成（约前 2-2.5 天）

这部分是交付底线，少一项都会显著扣分。

### 1. 数据建模 + 异步 Mock 层（基础设施）
- 产品、客户、持仓关系、跟进记录四张表的字段设计——这是 ★★★★★ 业务契合度最直接的体现
- MSW 或 json-server 模拟异步请求（题目明确要求不能直接 import 数据)
- 产品 ≥ 8、客户 ≥ 8,且数据要"可信"（真实感的基金名/客户名/金额分布）

### 2. 产品货架（模块一核心）
- 列表 + 按类型/状态筛选 + 按名称搜索
- 详情查看（弹窗或侧栏最省事，够用就行）
- 字段至少覆盖：名称、类型、最新净值、成立规模、状态

### 3. 客户管理（模块二核心）
- 列表 + 基本信息
- **客户-产品持仓双向关联**（题目特别点名的需求："某个客户买了哪些产品 / 某个产品被哪些客户持有"——强烈建议产品详情里也能看到持有人列表，形成闭环）
- 跟进记录的查看与新增（表单即可）

### 4. 数据概览（模块三核心）
- 2 个图表就够，但要"对销售有价值"，例如：
  - 在售产品按类型的规模占比（饼图/环形图）
  - 客户认购金额 Top N 或近 N 个月新增跟进趋势
- 图表内容的选择本身就是业务理解力的体现，别选"产品数量分布"这种没信息量的

### 5. DESIGN.md 骨架
- 权重和代码同等——必须写，不是交付前最后拼的
- 建议开工当天就起草，边做边补
- AI 协作日志要求"至少 2 个完整 prompt→输出→决策"过程，必须实时记录，事后编不出来

---

## P1：强烈建议做（第 3 天）

这部分不是硬要求，但 ROI 很高，不做会让整体显得"只是功能能跑"。

### 1. AI Agent 深度集成（⭐ 加分项，但几乎是拉开差距的关键）
- 题目明确说不要"套一个聊天框"——选 **1 个**方向做透比做 3 个浅的强
- 推荐优先级：
  - **智能查询（性价比最高）**：自然语言 → 结构化查询 → 直接渲染到现有列表/详情组件。能复用现有 UI，不用新造轮子
  - **辅助录入**：跟进记录的自然语言解析填表。场景真实、演示效果好
  - 自然语言驱动 Dashboard 最惊艳但实现成本最高，时间紧不建议
- Mock LLM 完全可接受（题目明说），但要把"Agent 与业务系统的交互架构"在 DESIGN.md 里讲清楚

### 2. 边界状态
- Loading / 空 / 错误三态——题目在「产品体验 ★★★★」里明确提了
- 成本很低、体感差距很大

### 3. 视觉与交互打磨
- 选一个成熟 UI 库（Ant Design / shadcn），别自己手搓
- 信息层级（表格密度、留白、主次对比）

---

## P2：时间富裕再动（第 4 天前半）

- **后端服务**：题目说"非必须"，Mock 层已经够用。除非你全栈很熟且剩半天以上，否则不要为了加分去搭，极易烂尾
- 第二个 Agent 能力方向
- 更多图表 / 更花哨的图表交互
- 导出、批量操作、权限等"企业感"功能

---

## P3：明确不做（但 DESIGN.md 里要写）

题目专门问"重要但没做的，再给 2 天会优先做什么"——这一问就是在考取舍判断。**主动砍功能、在文档里解释为什么砍**，比硬塞半成品得分高。典型可写：

- 真实后端（RBAC、审计日志）
- 移动端适配
- Agent 的多轮对话与上下文记忆
- 数据可视化的钻取联动

---

## 时间切分建议

| 时间 | 目标 |
|------|------|
| Day 1 | 技术选型定稿、数据建模、Mock 层、布局骨架、DESIGN.md 起草 |
| Day 2 | 产品货架 + 客户管理 P0 全部跑通 |
| Day 3 | Dashboard 2 图 + 边界状态 + Agent 一个方向打通 |
| Day 4 | 打磨 + DESIGN.md 收尾 + 部署 Demo + 录屏兜底 |
